# 需求管理及分析工作指南精读

## Source

- 原始文件归档：`docs/source-extracts/original/requirements-management-analysis-guide.doc`
- 抽取文本：`docs/source-extracts/requirements-management-analysis-guide.txt`
- 文档性质：金融系统需求管理/需求分析工作指南，来自长期需求组经验。

## 核心理解

这份文档表面上是需求分析规范，实质上是一套金融系统防事故方法论。

它的核心不是“把用户说的话写成文档”，而是：

```text
识别真实业务问题
-> 区分用户需求和系统需求
-> 控制范围和方案复杂度
-> 把需求写到可评审、可测试、可追踪
-> 把异常、安全、性能、环境提前纳入
-> 通过确认和变更记录降低后续扯皮与生产事故
```

这和 A9 当前要做的 MoE 评审高度相关：评审器不能只看代码是否通过测试，
还必须判断任务是否解决真实问题、是否过度设计、是否可验证、是否遗漏异常和安全边界。

## 第一层：需求审核不是收需求，而是判定问题边界

原文在 4.1 要求收到需求后先审核：

- 需求背景和目的。
- 用户要解决什么问题。
- 是新功能、功能变更还是功能补充。
- 菜单/模块/路径是否准确完整。
- 是否涉及多模块、多系统、后台表定义、数据维护。
- 是否需要负责人确认和跨部门汇签。
- 多系统需求是否要拆分，并建立强关联关系。

A9 解读：

- 任何 worker 任务都不能只看“用户要我做什么”，还要判断它属于哪类任务。
- 多系统/多模块任务必须拆分，不能一个 worker 一口吃掉。
- 如果任务之间有强依赖，必须显式建关联，而不是靠聊天上下文记忆。

落到 MoE：

- `scope_expert` 必须判断任务是否需要拆分。
- `product_mainline_expert` 必须判断本任务和当前主线的关系。
- `dependency_expert` 必须判断是否存在同步上线/前置依赖/跨模块联动。

## 第二层：70%-80% 是沟通，说明需求本质是对齐，不是执行

原文 5 章强调需求分析中 70%-80% 是沟通，并要求先明确总体方向，再逐步确认细节。

这对 A9 的含义：

- 24 小时 worker 是执行机器，不适合自行决定方向。
- supervisor/monitor 的价值不只是检查错误，而是维持方向一致。
- 缺方向的任务不应该直接实现，应进入需求澄清或方案比较。

落到 MoE：

- `communication_expert` 检查任务是否包含背景、目标、边界、成功标准。
- 如果缺少“为什么”，不能只靠 worker 自己补脑。

## 第三层：沟通前准备等价于 A9 的 reference_scan

原文 5.1 的准备包括：

- 参阅现有需求文档。
- 系统测试，感受现有功能。
- 代码排查，理解后台逻辑。
- 现场体验。
- 准备问题列表，站在用户角度问问题。

A9 的“抄抄抄”应该补上这个前置纪律：

```text
不是直接抄项目
而是先理解现状、用户、系统、代码、参考机制、问题列表
```

之前 worker broad scan 爆预算，正是因为只有“看参考项目”的口号，没有给出问题列表和现状锚点。

落到 MoE：

- `research_preparation_expert` 检查是否先有 repo map、现状证据、参考项目精确入口。
- 没有入口时，先由 monitor 做轻量定位，再交给 worker。

## 第四层：区分“是什么”和“为什么”是最高优先级

原文 5.3 是整份文档的灵魂：

- 用户经常直接提出实现方案。
- 如果不理解背景，可能做完也不能解决真实问题。
- 必须区分需求背景和实现方案。
- 必须搞清楚必须做、应该做、可以做。
- 不是所有需求都要系统实现。
- 系统实现贡献低、成本高、风险大时，应建议系统外处理、另辟新径或放弃。

这是 A9 不能跳过的核心。

对当前项目的直接校正：

- “做 mobile 页面”是方案，不是需求本身。真实需求是手机接管主控入口，避免人被桌面绑住。
- “做 24 小时机器”是方案，真实需求是持续执行、不中断、不丢因果、能被监控纠偏。
- “做 MoE score”也是方案，真实需求是多视角评审防止 worker 偏航。

落到 MoE：

- `why_expert` 必须对每个任务输出：
  - 真实问题是什么。
  - 当前方案是否能解决。
  - 是否有更低成本方案。
  - 是否属于必须/应该/可以。
- `why_expert` 有否决权：如果任务只有方案没有问题，必须退回澄清。

## 第五层：系统需求是用户需求的翻译，不是照抄

原文 5.6 把用户需求和系统需求分开：

- 用户需求从业务角度描述想实现什么。
- 系统需求站在系统角度描述系统应该是什么样。
- 用户、需求分析、开发三方讨论后达成共同认可。
- 系统需求要明确功能范围、功能点、风险点、扩展性、可靠性、性能。

A9 对应：

```text
用户原始想法
-> 需求精读/因果变迁
-> A9 系统需求
-> 任务切片
-> worker 执行
```

不能把用户一句“继续抄抄抄”直接变成代码任务。

落到 MoE：

- `system_requirement_expert` 检查任务是否已经从“业务/产品表达”翻译成“系统行为”。
- 输出必须包含范围、功能点、风险、非功能。

## 第六层：方案选择是 tradeoff，不是单点最优

原文 5.6 和 5.7 通过方案优缺点表选择方案，并提出原则：

- 减少模块间耦合。
- 减少跨系统耦合。
- 按岗位职责切分功能。
- 优先解决基本诉求。
- 降低系统实现复杂度，避免过度设计。
- 页面、批处理、接口、ETL、数据维护各有适用场景。

A9 对应：

- Redis/Rust/Tailscale/tmux/WebSocket/HTTP/API 不是信仰选择，要看场景。
- worker 任务也要按职责切分：需求、机制抽取、实现、测试、记录不能混乱。
- 先做最小可验证闭环，再扩展。

落到 MoE：

- `architecture_expert` 不能只说“架构好/不好”，必须输出方案比较：
  - 耦合度。
  - 实现复杂度。
  - 风险。
  - 可扩展性。
  - 当前是否过度设计。
- `role_boundary_expert` 检查是否把前一岗位的职责塞到后一岗位，类比文档里的“审核环节不能修改录入账单”。

## 第七层：需求规格不是文档形式，而是可交付合同

原文 6 章要求说明：

- 业务背景及调整原因。
- 业务流程，包括系统内和系统外。
- 测试要点，从业务角度确认重点。
- 风险提示。
- 方案类型对应的细化程度。
- 页面字段、按钮、校验、报错。
- 批处理抽取范围、频率、优先级。
- 接口调用场景、服务方、调用方、输入/输出、必填、码表。
- 数据维护范围、对象、属性、顺序。
- 通知对象、主题、正文、附件。
- 编码规则字段、长度、总长度。

A9 对应：

- 一个任务如果要实现 API，必须说明调用场景、输入/输出、错误和测试。
- 一个任务如果要实现后台任务，必须说明频率、优先级、日志和异常。
- 一个任务如果要实现页面，必须说明字段、按钮、状态、校验、报错。

落到 MoE：

- `spec_contract_expert` 检查任务是否有足够细化程度。
- `test_expert` 检查测试要点是否来自业务关注点，而不是只覆盖代码分支。

## 第八层：质量指南是 MoE 的核心评价维度

原文 7 章列出需求质量标准：

- 无歧义性。
- 可读性。
- 完整性。
- 可验证性。
- 一致性。
- 可修改性。
- 可追踪性。
- 异常处理流程。

这些比“score 0.7”更重要。A9 的 MoE 应改成质量维度判定：

| 质量项 | A9 评审问题 |
| --- | --- |
| 无歧义 | 任务中的对象、动作、成功标准是否明确 |
| 可读 | worker prompt 是否结构化、短句、分段 |
| 完整 | 是否覆盖正常流、异常流、历史数据/兼容性 |
| 可验证 | 是否有可执行检查和量化标准 |
| 一致 | 是否与项目主线、已有 API、术语、测试框架一致 |
| 可修改 | 是否避免冗余、是否局部化、是否易演进 |
| 可追踪 | 是否能从需求 -> 任务 -> diff -> test -> evidence |
| 异常 | 是否有失败处理、重试、通知、人工介入 |

## 第九层：非功能不是附属，而是金融系统接受条件

原文 8 章强调非功能必须量化和测试，包括：

- 易用性。
- 性能响应、容量、可靠性、可伸缩性。
- 操作环境和设备。
- 安全：账号密码、权限、输入检测、审计、敏感数据输出。

A9 的通信/Agent OS 方向尤其需要吸收：

- 手机端必须明确目标设备和网络环境。
- 通信必须有断链、重连、超时、恢复、审计。
- 远程主控入口必须有权限和审计。
- 敏感日志和 session 不能随便输出到页面或 worker prompt。

落到 MoE：

- `nfr_expert` 检查性能、稳定性、环境。
- `security_expert` 检查权限、输入、审计、敏感数据。
- 对高风险项有 hard gate。

## 第十层：异常处理是金融系统经验里最值钱的部分

原文多次强调异常：

- 多系统交互时，异常要能重发或通知。
- 异常不能导致无人知晓或无法继续处理。
- 异步按钮要提示后续在哪里查看结果。
- 关键交易应记录历史，可回溯回退。

A9 目前真正要抄的不是某个 UI，而是这套异常治理心智：

```text
任何无人值守任务
必须有：
失败分类
重试策略
人工介入点
证据路径
恢复方法
审计记录
```

这正好和我们刚做的 reconnect governance、worker budget、scope guard、policy attestation 对齐。

## 对 A9 MoE 评审器的修正

刚才的 `a9_monitor.py` 只有雏形，必须升级为需求方法论驱动的专家组。

建议专家如下：

1. `why_expert`
   - 判断真实问题、背景、目的。
   - 区分需求和方案。
   - 有否决权。

2. `scope_dependency_expert`
   - 判断模块/系统/任务拆分。
   - 判断强依赖和同步关系。

3. `system_requirement_expert`
   - 将业务/产品表达翻译成系统行为。
   - 检查范围、功能点、风险、非功能。

4. `tradeoff_architecture_expert`
   - 比较方案优缺点。
   - 检查耦合、复杂度、可扩展性、是否过度设计。

5. `role_boundary_expert`
   - 检查人类/monitor/worker/系统模块职责是否混淆。

6. `test_verifiability_expert`
   - 检查可量化、可测试、测试点是否来自业务关注点。
   - 对数据敏感任务，必须看 schema/table/state/event 是否被验收。

7. `quality_expert`
   - 检查无歧义、可读、完整、一致、可修改、可追踪。

8. `exception_governance_expert`
   - 检查异常流、重试、通知、恢复、审计。

9. `nfr_security_expert`
   - 检查性能、容量、环境、权限、输入检测、敏感数据、审计。

10. `execution_governance_expert`
    - 检查 worker 外显行为：scope、commands、tests、license、token、evidence。

11. `product_mainline_expert`
    - 检查哲学/业务逻辑/主线是否大于工程实现。

12. `external_learning_expert`
    - 检查需要抄和考证时是否真的看了参考项目或外部资料。

13. `product_pressure_expert`
    - 检查是否有推翻弱方案、压缩范围、提高验收标准的能力。

14. `data_model_expert`
    - 检查数据、表结构、状态、事件是否映射真实业务结构。
    - 数据结构对了，业务大概率对；但仍要覆盖权限、流程、异常和时序。

15. `performance_depth_expert`
    - 检查性能、稳定性、预算和压测边界是否体现产品厚度。

## 合议规则

不能平均分。

合议应按金融需求评审逻辑：

```text
hard gate:
  安全/权限/敏感数据/异常无人知晓/无测试/无真实问题 -> block

tradeoff gate:
  方案复杂度高、耦合高、收益不明 -> monitor_review 或 needs_tradeoff

execution gate:
  worker 越权、爆 token、未声明检查、缺 envelope -> repair/narrow

progress gate:
  主线推进明确、证据完整、测试通过 -> continue
```

## 对当前项目的直接影响

1. 不能再说“MoE 评分”就算方法论完成。
2. `a9_monitor.py` 应从行为违规检测升级为需求评审委员会。
3. worker prompt 要先带：
   - 背景。
   - 真实问题。
   - 系统需求。
   - 必须/应该/可以。
   - 测试要点。
   - 风险和异常。
4. 自动化不能只看 pass/fail，要看是否解决真实问题。
5. 对手机端、远程控制、多机器接入，必须补非功能、安全、异常、审计需求。

## 二次精读后的遗留核心问题

当前 A9 已经补上“先定形，再治理”，但对照原文仍有几个核心缺口。
这些缺口不是工程 gate，而是任务进入 worker 前的需求分析必填项。

### 1. 需求调整和变更留痕还不够

原文 4.5 强调：需求在用户确认、开发、测试过程中发生问题或变更时，
必须分析评估、标注修改点、通知相关角色、重新确认。

A9 目前有 session causal memory 和 git commit，但还没有把每次任务口径变化写成
稳定的“需求变更记录”：

- 为什么从旧任务切到新任务。
- 哪个旧判断失效。
- 本次变更影响哪些任务、测试、文档和 worker prompt。
- 谁确认放行，或为什么由监控者临时放行。

后续 A9 的 `memory_commit` 不应只记录代码事实，还要记录需求变更事实。

### 2. “本次不做什么”必须显式确认

原文完整性要求：用户需求中最终沟通下来本次不做的内容，要剔除或列出原因并确认。

A9 现在经常写“下一步做什么”，但“不做什么”不够硬。
这会导致 worker 把 mobile、MoE、session、通讯、金融模型混在一起。

每个进入 worker 的任务都必须带：

```text
out_of_scope:
- 本次不做什么
- 为什么不做
- 以后什么条件下再做
```

### 3. 术语表和对象语言缺失

原文 7.1/7.5 反复强调术语一致，尤其要避免同一对象多个名称。

A9 目前有这些混用风险：

- 24h worker / 执行机器 / worker / automation。
- 主控 / monitor / supervisor / operator。
- session / raw session / runtime session / memory commit。
- gate / guard / monitor score / MoE / role review。

这会直接污染任务切片和测试验收。需要一个小型术语表，至少放进
`docs/project.md` 或独立 `docs/glossary.md`。

### 4. 方案类型细化程度没有成为任务模板

原文 6.5 按方案类型定义细化要求：

- 页面：字段、按钮、校验、报错、前后台处理。
- 批处理：抽取范围、频率、优先级、流程、日志。
- 接口：调用场景、服务方、调用方、输入输出、必填、码表、规则。
- ETL：同步范围、时效性、频率、上下游。
- 数据维护：对象、属性、范围、顺序。
- 通知：接收人、主题、正文、附件、模板验证。
- 编码规则：字段取值、长度、总长度。

A9 现在的任务模板没有先判断“这是什么方案类型”，所以 worker 容易用同一套
工程语言处理所有任务。后续任务成形卡必须先选 `solution_type`，再按类型补字段。

### 5. 历史数据/历史状态维护没有被前置

原文完整性要求：涉及历史数据维护时，必须提供具体维护方案，不能只写“需要维护”。

A9 对应的是：

- 已有 `.a9/runs`、`.a9/tasks`、goal state、memory commit 是否需要迁移。
- 已有 mobile/control API response 是否兼容。
- Redis/MySQL 里的旧 key、旧 schema、旧 event 是否需要升级。
- 旧 session 精读和 causal memory 是否需要补索引。

这类内容不能等测试失败后再补，需求分析阶段就要说明。

### 6. 可验证性必须量化，但不要过早数字化 gate

原文要求可验证和可量化，但也强调量化来自需求分析和沟通确认。

A9 的错误是：先拍脑袋设 token、行数、score 阈值，再让它阻断执行。
正确做法是：

```text
先明确业务/系统接受条件
-> 再定义观测指标
-> 运行几轮收集分布
-> 最后把稳定指标升级成 gate
```

所以早期数值只能是 observation，不是 hard gate。

### 7. 用户确认在 A9 里对应“监控者确认”

原文有评审、用户确认、需求调整邮件通知。

A9 没有传统用户确认流程，但必须有等价物：

- 监控者确认任务成形卡。
- 监控者确认变更记录。
- 监控者确认“不做项”。
- 监控者确认 worker 产物是否进入下一轮。

手机/control plane 后续应该服务这个确认流程，而不是只做运行状态页面。

## A9 任务进入 worker 前的最小需求分析卡

每个非平凡任务进入 24h worker 前，至少应有：

```text
problem:
  真实问题是什么，不是方案名。
why_now:
  为什么现在做。
must_should_could:
  must / should / could 分层。
system_requirement:
  系统应该表现成什么行为。
solution_type:
  page / batch / api / etl / data_maintenance / notification / coding_rule / runtime_infra。
data_shape:
  数据、状态、事件、对象、属性。
normal_flow:
  正常流程。
exception_flow:
  异常、重试、通知、人工介入、审计。
acceptance:
  可验证验收点，测试点来自业务关注点。
out_of_scope:
  本次不做什么，以及原因。
reference_entry:
  参考项目和具体入口，不是泛泛“看顶级项目”。
change_record:

  worker 允许读什么、改什么、跑什么检查。
```
