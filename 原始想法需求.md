## 一句话

A9 要做的不是普通量化模型，也不是普通类 Codex CLI。

A9 是一个以交易哲学为宪法、以成熟项目复制为工程方法、以数据验证为裁判、
以 24 小时 Agent Runtime 为执行机器、最终沉淀私有金融交易工程模型的
Agent OS / ResearchOps 系统。

核心公式：

```text
交易哲学
-> 交易逻辑
-> 风险边界
-> 数据验证
-> 最小策略闭环
-> 工程架构
-> TDD / 压测 / 监控
-> 小资金实盘
-> 归因优化
-> AI 辅助迭代
```

工程公式：

```text
找对标
-> 抽机制
-> 能抄绝不手搓
-> 多项目融合
-> 小步魔改
-> Diff / Git / Sandbox 管住
-> 数据验证
-> 压测
-> 监控
-> 轨迹入库
-> 训练/蒸馏私有模型
```

## 最高原则

### 1. 哲学优先

交易系统不是先写代码，而是先有交易哲学。

哲学不是口号，它决定：

- 什么收益不能要。
- 什么回测不能信。
- 什么策略不能下水。
- 什么风控不能绕过。
- 什么 patch 必须拒绝。
- 什么数据属于污染。
- 什么异常必须停机。

所以 A9 未来需要 `TRADE_AGENTS.md` 或同等宪法文件，写清：

- 交易哲学。
- 风控红线。
- 数据验证标准。
- 回测无效条件。
- 禁止自动下水的条件。
- 策略从假设到小资金实盘的流程。

### 2. 抄抄抄是核心方法

原始想法里反复出现的关键句：

```text
找对标，组合，能抄袭绝对不写。
只给目标，只允许抄，然后拼。
必须数据验证，其他不管。
```

这不是偷懒，而是复杂系统最务实的方法：

- AI 从零发明复杂逻辑会幻觉。
- 顶级治理方案大概率已经在成熟项目、头部公司、开源框架或论文里出现过。
- A9 的优势不是“凭空创造”，而是发现信息差、找到顶级对标、抽出核心机制、
  写胶水、用数据验证。

抄的对象不是一段代码，而是：

- 状态机。
- 失败模式。
- 边界条件。
- 重试/回滚/审计/限流/熔断。
- Diff 纪律。
- Repo map。
- Sandbox harness。
- 交易系统事件流。
- 风控和数据验证流程。

### 3. 数据验证第一，压测第二

AI 的解释、置信度、自评都不是裁判。

真正的裁判是：

- 编译器。
- 单测。
- 集成测试。
- 回测。
- replay。
- 延迟统计。
- 滑点/手续费/冲击成本。
- 风控检查。
- 小资金实盘结果。
- 归因报告。

压测第二的意思是：先验证交易逻辑和数据真实性，再验证系统压力、延迟、稳定性。
没有数据验证的压测只是工程自嗨。

## 原始产品脉络

### 阶段 1：类 Codex CLI / Rust UNIX 客户端

最早的问题是：

```text
有没有办法做一个 codex cli 一样的 unix 客户端，也是抄抄抄模式？
```

早期设想：

- 做一个平台专属 UNIX CLI。
- 能生成 shell 命令并确认执行。
- 能读取本地代码。
- 能输出 Diff / SEARCH-REPLACE。
- 能接平台模型网关，消耗平台 token。
- Rust 优先，因为单文件、低延迟、无 Node/Python 环境依赖。

参考对象：

- Codex CLI：agent loop、工具调用、sandbox、approval、配置和 session。
- Aider：repo map、SEARCH/REPLACE、diff、architect/editor、Git 闭环。
- Claude Code：终端内理解项目、多文件修改、权限控制。
- aichat：Rust CLI、多模型 provider、REPL、shell assistant、tools/agents。
- SWE-agent：issue -> shell/git/docker -> patch -> test。
- Continue/Cline/Roo/OpenHands/opencode：客户端、事件流、工具边界。

关键判断：

普通聊天模型老是整文件重写；真正的 coding agent 必须让模型只输出结构化编辑，
本地 deterministic apply，再用 git/test/guard 判定。

### 阶段 2：Trading-native AI Dev Agent

`codex.md` 把目标从“CLI 工具”升级为：

```text
Trading-native AI Dev Agent
```

它不是普通 coding assistant，而是：

- AI 工程操作系统。
- 交易研究助理。
- 代码修改器。
- 验证流水线。
- 数据采集器。
- 交易工程模型训练数据工厂。

它的工作流不是“用户问一句，AI 改一段代码”，而是：

```text
交易假设
-> 数据检验
-> 代码实现
-> 单测
-> 回测
-> 压测
-> 风控检查
-> 模拟盘/小资金
-> 归因
-> 人类确认
-> 记录为训练数据
```

训练交易模型之前，必须先有这个工具和工作流。否则训练出来的是会说交易黑话的
幻觉模型，而不是能修 bug、跑回测、识别未来函数、生成可应用 diff 的工程模型。

### 阶段 3：24 小时执行机器

讨论中发现：如果每次都人工让 AI 去找对标、抄机制、改代码、跑测试，效率太低。
所以出现了 24 小时执行机器。

它的定位：

- 不是产品经理。
- 不是业务战略家。
- 是执行机器。

它负责：

```text
reference_scan
-> mechanism_extract
-> vendor_import
-> implement
-> test
-> record
-> repair
```

人类/主监控负责：

- 方向。
- 架构判断。
- 业务讨论。
- 任务拆解。
- 监控。
- 验收。

24 小时机器必须能跑，但不能自由失控：

- 每个任务 isolated worktree。
- 不直接改主分支。
- Patch 必须过 guard。
- 测试失败进入 repair。
- token/log 增长过快要停。
- 真实 worker 质量差时由主监控接管。

### 阶段 4：Session 精读治理

真正的痛点不是“上下文不够长”，而是：

```text
长 session 中的原始意图、因果变迁、关键细节会退化。
```

原始想法里明确区分了两类 session：

- 外部 Codex/operator session：人类和主监控窗口产生的 JSONL，用于恢复原始意图、
  精读、提取 doctrine。
- A9 runtime session：24 小时机器产生的 task/run/flow/evidence，用于执行治理。

二者不能混存。

session 精读不是普通 summary。它要每 3 轮或 10 轮提取：

- 原始意图。
- 触发事件。
- 决策变迁。
- 被废弃的想法。
- 被保留的关键细节。
- 当前有效主线。
- 证据指针。

抗退化方案：

- Git-like memory commit：追加，不覆盖。
- keep / add / deprecate / supersede / repair 显式生命周期。
- raw event 永久保留。
- 代码/日志用物理快照指针，不让 AI 总结掉细节。
- 静态约束隔离，不参与递归压缩。
- compact summary 只是交接材料，不是事实源。

这套逻辑后来变成 A9 的 `session_refresh -> session_close_reading` mini-flow。

### 阶段 5：Agent OS / 手机控制面

`app.md` 带来的关键启发是：

```text
手机不是生产端，而是 control plane。
host/server 才是 compute plane。
```

官方 Codex mobile / automation / long-horizon memory 的讨论，触发了 A9 的 Agent OS 形态：

```text
手机/浏览器/CLI
-> control plane
-> operator session
-> run/flow/status/diff/check/token
-> approve/reject/retry/pause/continue
-> host 上 tmux/Codex/A9 runtime 真正执行
```

UI 方向不是 Jupyter，而是：

```text
70% ChatGPT
20% Linear/Jira
10% VSCode/terminal
```

原因：

- Agent workflow 本质是协作，不是 notebook cell。
- 手机高频操作是看、审批、追加任务、看 diff、看 replay 状态。
- 大 diff、大 coding、大数据分析留给桌面/host。

Tailscale / SSH / tmux 的意义：

- 让手机和服务器像内网一样连接。
- tmux 保持长 session 不死。
- 手机随时 attach 同一个 Codex/A9 session。
- 这是早期最稳 remote ops 形态。

### 阶段 6：通讯治理和多服务器接入

当 A9 走向 Agent OS，稳定性成为第一问题。

需求变成：

```text
多机器快速接入
断网/断链自动治理
用户对技术细节无感
极快、高并发、稳定、不丢事件
```

因此主线从 UI 细节切到通讯治理：

- Rust 做 gateway 和热路径。
- Redis Streams 做事件总线。
- Redis Functions 做原子状态转移。
- RedisJSON 做 hot snapshot。
- RedisTimeSeries 做健康和延迟指标。
- Tailscale/WireGuard 做私有网络基座。
- SSH/tmux 做 bootstrap 和兜底接管。
- Python 保留模型逻辑、prompt、业务策略。
- MySQL 做 canonical durable state。

Barter-rs、Codex、OpenClaw/Lobster、Aider、SWE-agent 都是可抄参考。

## 私有模型路线

原始想法不是“直接训练一个全能模型超过 GPT-5.5”，而是：

```text
专属金融交易工程小模型 + 专属 Agent 系统
在 A9 自己的交易工程任务上超过 GPT-5.5 裸模型。
```

### 正确顺序

```text
Teacher API / 大模型
-> A9 Agent Harness
-> 真实轨迹数据
-> SFT / DPO / GRPO / 蒸馏
-> 4090 可推理的小模型
-> 金融交易工程 Agent
```

不能反过来。

没有工具、工作流、验证、轨迹数据，就没有好训练数据。

### 要训的不是一个能力，而是四类能力

- 仓库理解力：repo map、symbol graph、相关文件、上下文治理。
- 工程执行力：Diff、SEARCH/REPLACE、测试失败修复、最小 patch。
- 交易风控判断力：拒绝未来函数、回测污染、成本忽略、风控绕过。
- 工具调用与验证能力：能调用测试、回测、压测、日志、数据审计。

### 不把源码硬塞进权重

成熟项目用于抽机制，不是让模型背代码。

交易项目参考：

- Hummingbot：connector、策略配置、CEX/DEX 接入、做市/套利/执行结构。
- NautilusTrader：event-driven core、research-to-live parity、订单状态机、
  deterministic simulation、Rust core + Python control。
- Barter-rs：Rust live/paper/backtest、高性能事件系统、external commands。
- Lean/QuantConnect、ccxt、Polars、DuckDB、Tokio、Rayon 等可作为后续参考池。

训练素材应该变成：

```text
成熟源码/文档
-> teacher 精读
-> 工程不变量
-> 反例
-> patch 任务
-> 异常治理题
-> 架构选择题
-> A9 自己仓库真实任务轨迹
-> 通过验证的样本
```

### TimesFM / 时间序列模型的位置

TimesFM 或类似时序模型不应该把原始价格序列塞给 LLM。

正确做法：

```text
TimesFM / 时序模型
-> regime / trend / confidence / key_level / anomaly 等结构化特征
-> 交易研究模型 / 风控模型 / 策略验证流程
```

交易 LLM 管逻辑、解释、风险、策略假设和工具调用；时序模型管数字预测和模式。

### 4090 的定位

4090 不是用来硬塞 385B/400B 巨兽推理。

正确路线是：

- 大模型/Teacher 生成高质量轨迹、问答、反例、patch 数据。
- 学生模型做 14B/32B/30B-A3B 级别。
- 通过 LoRA/QLoRA/蒸馏/偏好训练，把 A9 的“抄机制 + Diff + 交易验证”训练成肌肉记忆。
- 4090 负责低延迟本地推理、快速试错、工具调度、小模型 judge/editor。

## 复杂项目方法论

原始讨论中多次推翻“AI 自动解决复杂项目”的幻想。

最后稳定下来的方法是：

```text
复杂项目不是让 AI 自由发明。
复杂项目 = 找顶级对标 + 抽机制 + 胶水拼装 + 数据验证。
```

关键不是“AI 有多聪明”，而是“能否发现顶级项目在哪里”。

信息差就是护城河：

- 顶级项目在哪里。
- 哪个模块是真的核心。
- 哪些失败模式别人已经治理过。
- 哪些跨行业方案可以迁移。
- 哪些源码能抄，哪些只能抄机制。

看大型项目的流水线：

```text
repo map
-> 找入口
-> 看测试
-> follow the money / follow the order
-> 抽调用链
-> 抽状态机
-> 只拿核心切片
```

AI 的强项不是从零创造，而是：

- 信息差搜索。
- 大代码库阅读。
- 抽象机制。
- 格式转换。
- 写胶水。
- 根据报错修补。

AI 的弱点必须被工程管住：

- 不让它直接写大段核心逻辑。
- 不让它直接改全文件。
- 不让它自评通过。
- 不让它越过测试和风控。

## A9 当前主线牵引

这份原始需求对后续 session 精读和 24 小时机器有一个主线牵引：

```text
A9 的任何任务都要问：
1. 它服务的是交易哲学和数据验证闭环吗？
2. 它是在找对标、抽机制、抄成熟治理，还是在手搓？
3. 它是否保留了因果变迁和原始细节？
4. 它是否让 24 小时执行机器更稳定、更可监控、更低 token？
5. 它是否让未来私有交易工程模型产生更高质量轨迹数据？
6. 它是否经过测试、回测、压测、风控或其他客观验证？
```

如果答案是否定的，要么暂停，要么降级为支线。

## 对自动化精读的归因规则

后续 `session_refresh / session_close_reading / memory_commit` 不只是总结聊天。
它必须按这条主线做归因：

### 归因类别

- `original_intent`: 原始想法，来自本文或归档原件。
- `product_evolution`: 产品形态演进，例如 CLI -> 24h runtime -> Agent OS。
- `engineering_constraint`: 工程硬约束，例如 Rust/Redis/Git/Diff/Sandbox。
- `session_governance`: session 精读、compact drift、memory commit。
- `reference_copy`: 发现新对标或修正对标优先级。
- `expired_branch`: 已过期支线，例如过度页面细节、只靠页面监控。
- `open_question`: 仍未证实的奇思妙想，例如私有模型训练路径、TimesFM 融合。
- `validated_doctrine`: 已被代码/测试/运行证据验证的规则。

### 每批精读必须保留

- 原话位置：source file / turn / line。
- 原始意思：不要过度润色。
- 变迁原因：为什么从 A 改到 B。
- 当前状态：active / deprecated / superseded / open。
- 证据指针：文档、代码、测试、run、commit。
- 下一步：交给 24 小时机器的 bounded task。

### 不允许

- 不允许只写“总结一下”。
- 不允许把奇思妙想过滤成普通项目管理语言。
- 不允许把已经废弃的想法当当前主线。
- 不允许让 UI、模型训练、量化策略、通讯治理互相混成一坨。
- 不允许把 compact summary 当事实源。

## 当前结论

A9 当前应该先继续补强：

```text
24h Agent Runtime
-> Session Governance
-> Communication Governance
-> Diff/Git/Sandbox/Guard
-> Copy Pipeline
-> Evidence/Trajectory Store
```

这些不是偏离原始交易模型梦想，而是在给最终的金融交易工程模型铺地基。

最终目标仍然是：

```text
用 A9 作为 ResearchOps / Agent OS
持续生产高质量工程轨迹、交易研究轨迹、风控审计轨迹
再训练/蒸馏出私有金融交易工程模型
在自己的交易工程闭环里超过通用 GPT-5.5 裸模型
```
